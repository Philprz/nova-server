## 2. Catégorie (A) — correctifs vérifiables à GARDER

Pour chacun, l'intention est certaine (import manifestement absent, ou bug à intention non ambiguë).

- **A1 `datetime`** : utilisé une seule fois (`self._cache_loaded_at = datetime.now()`) sans aucun import dans le module → NameError garanti sur ce chemin. `from datetime import datetime` est le seul sens possible.
- **A2 `asyncio` (bloc `__main__`)** : n'impacte que l'exécution CLI directe du fichier (`if __name__ == "__main__"`). Comportement runtime serveur strictement inchangé.
- **A4 `datetime` (pricing_validation)** : `datetime.now().isoformat()` en l.457 dans `update_decision_price` (routeur monté `/api/validations`), alors que le seul `import datetime` était local à une **autre** fonction (`get_validations_by_case_type`, l.332) → NameError latent. Import module-level = correction certaine.
- **A5 `EnhancedDevisWorkflow`** : `EnhancedDevisWorkflow(...)` utilisé en l.916 (`_execute_quote_generation`) ; classe définie dans le même module déjà importé. Import manifestement manquant.
- **A6 `call_sap`** : `await call_sap(...)` en l.190 dans `check_product_exists` (monté `/api/sap`), import local présent dans une autre fonction (l.88) mais pas celle-ci. Ajout = correction certaine.
- **A7 `time`** : `time.time()` (devis_workflow l.9363) sans `import time` au module (seulement des imports locaux ailleurs) → NameError sur ce chemin. Import stdlib manifestement requis.
- **A8 imports `LLMExtractor` / `client_creation_workflow`** : symboles existants (services.llm_extractor, workflow.client_creation_workflow) utilisés sans import. Manifestement manquants. *(NB doublon initial `get_llm_extractor, get_llm_extractor` corrigé dans le même commit.)*
- **A10 `sap_data`→`sap_client_data` & `sap_results`→`result`** : le dict réellement construit juste au-dessus s'appelle `sap_client_data` ; la variable assignée par l'appel est `result`. Intention non ambiguë. **Couvert par le test** `test_create_validated_client_chemin_succes`.
- **A11 `total_weight_kg`** : `Product` possède `weight_kg` + `quantity` ; la trace affiche « Poids total ». La somme pondérée est l'unique sens. **Couvert par le test** `test_optimize_transport_calcule_le_poids_total`.
- **A12 `sap_list_draft_quotes`** : fonction réellement définie (`sap_mcp.py:950`) et déjà importée en lazy par `routes/routes_devis.py:98` selon le même pattern. Correction sans ambiguïté.

**Cas limite A3 (`get_mcp_connector()`)** : techniquement une reconstruction (choix `get_mcp_connector()` plutôt que `MCPConnector()`), mais le nom de variable d'origine était `mcp_connector` = le singleton, et la branche `else` voisine montre le pattern « obtenir un connecteur puis `.call_mcp` ». Intention fortement contrainte → **GARDER**. *Atteignabilité* : `call_mcp_with_progress` est importée (`devis_workflow.py:17`) mais **aucun site d'appel** n'a été trouvé (`grep call_mcp_with_progress(` = vide) → chemin de fait non exercé ; correctif néanmoins fidèle et sans risque.

## 3. Catégorie (B) — reconstructions comportementales : atteignabilité, hypothèse, tests

### B1 — singleton `cache_manager = RedisCacheManager()`  →  **INJOIGNABLE**
- **Usage** : routes `generate_quote_optimized` / `continue_quote_after_interaction` décorées `@router_v2.post(...)` (devis_workflow.py:6858/6890).
- **Preuve d'injoignabilité** : `router_v2 = APIRouter()` est défini **indenté à l'intérieur de la classe `DevisWorkflow`** (l.6856) — c'est un attribut de classe, pas un routeur module. Il est exporté dans `__all__` (l.9627) mais **jamais importé ni `include_router`** nulle part (`grep router_v2` hors devis_workflow = vide). Le vrai `/continue_quote` vivant est `routes_intelligent_assistant.py:474` (sur `assistant_router`, monté).
- **Hypothèse de la reconstruction** : un store d'état workflow partagé entre requêtes est nécessaire (save→get) ; `RedisCacheManager` (fallback mémoire) le fournit.
- **Effet de bord réel** : l'instanciation s'exécute **à l'import** du module (tentative ping Redis + log, fallback mémoire). C'est le seul impact runtime ; la route consommatrice est morte.
- **Test** : aucun.

### Lecture d'ensemble
- **Aucune régression sur du code qui marchait** : tous les items (B) atteignables (B5, B6, B8, B9, B10, B11) portaient sur des chemins **déjà cassés** (NameError/UnboundLocalError garantis) — le commit les fait passer de « 500 systématique » à « fonctionnel reconstruit ». Le risque n'est donc pas une régression mais une **reconstruction inexacte** vs l'intention métier d'origine.
- **2 reconstructions à examiner en priorité** : **B9** (forme `clients_data` entièrement inventée) et **B5** (passage de « toujours `None` » à « renvoie un produit » sur le chemin produit chaud). **B6** et **B8** débloquent des chemins importants (validation devis, choix utilisateur) et sont fonctionnellement nécessaires, mais la forme exacte des données mérite confirmation.
- **4 items sur code mort** (B1, B2, B3, B4) : sans effet runtime (sauf B1 qui instancie un `RedisCacheManager` à l'import). Candidats à un nettoyage ultérieur (suppression du mort) plutôt qu'à conserver — mais leur retrait **réintroduirait l'échec de compilation Cython** : à traiter en supprimant le code mort, pas en revertant les seules lignes.
- **Build** : les 111/111 et la suite 474/0 restent valides ; rien dans cet audit ne le remet en cause.
